# Rita2API 注册后聊天请求轮换流程说明

## 1. 文档目的

本文档专门说明：

- 新注册账号进入账号池后，什么时候会被聊天请求用到
- `/v1/chat/completions` 和 `/v1/responses` 如何从账号池里选号
- 账号轮换、失败计数、扣点、健康检查之间如何联动
- 当前实现里有哪些关键分支和注意点

这份文档是对 `docs/注册整体流程.md` 的补充，重点不再讲“怎么注册”，而是讲“注册好的账号如何承接后续聊天流量”。

---

## 2. 一句话总览

新注册账号一旦成功入库并通过基本验活，就会立刻成为账号池候选；后续聊天请求进入 `/v1/chat/completions` 或 `/v1/responses` 时，会通过 `AccountManager.next()` 从**已启用、点数大于 0、失败次数未超阈值**的账号中按轮转顺序挑一个账号，再把该账号的 `token` / `Cookie` / `visitorid` 拼成上游请求头发给 Rita。

---

## 3. 从“注册成功”到“进入可选账号池”

### 3.1 注册成功后的入库动作

自动注册主链成功后，`auto_register.auto_register_one()` 会继续把账号写入 `AccountManager`：

- 账号名：`auto-邮箱前缀`
- 保存字段：
  - `token`
  - `email`
  - `password`
  - `mail_provider`
  - `mail_api_key`
- 默认入库状态：
  - `enabled=1`
  - `quota_remain=100`

也就是说，只要成功入库，这个账号天然就具备进入轮换池的基础资格。

### 3.2 注册后立即验活

账号写入后，会立即调 `AccountManager.test_account()` 对新号做一次基本连通性测试：

- 调上游 `/aichat/categoryModels`
- 能正常拿到模型目录，就视为 token 初步可用

这个动作的意义是：

- 避免“注册成功但 token 立刻不可用”的账号直接进入长期使用阶段
- 让账号尽快处于一个已知可用状态

### 3.3 对自动补号的反馈

账号入池后，`summary().active` 会增加。由于后台自动补号线程是按“活跃账号数”判断是否补号，因此新号入池会直接影响后续补号判断。

---

## 4. 能被轮换选中的账号条件

账号池选号逻辑的入口是：

- `AccountManager.next()`

当前代码里，候选账号必须满足：

1. `enabled=1`
2. `quota_remain>0`
3. `failures < 失败阈值`

其中失败阈值当前是：

- `_KEY_FAIL_THRESHOLD = 3`

注意：

- SQL 初筛只过滤 `enabled=1 AND quota_remain>0`
- `failures` 阈值是在 Python 里二次判断的
- `token_valid` 本身不是 `next()` 的直接筛选条件，但健康检查会在发现 401 后把账号 `enabled=0`，因此最终仍会被排除在外

---

## 5. 轮换算法本身

### 5.1 轮换核心

`AccountManager.next()` 的行为是：

1. 查询所有 `enabled=1 AND quota_remain>0` 的账号
2. 按 `id` 排序
3. 使用内部 `_robin_index` 做 round-robin 轮转
4. 逐个跳过失败次数达到阈值的账号
5. 返回第一个符合条件的账号

### 5.2 当前顺序的真实含义

由于当前 SQL 是 `ORDER BY id`，所以轮换顺序并不是按 `created_at`，而是按账号 `id` 字典序排序后再轮转。

这点意味着：

- 新注册账号不一定“最后才被轮到”
- 也不一定严格按注册先后顺序服务请求
- 实际顺序取决于 `id` 的字典序以及当前 `_robin_index` 所处位置

### 5.3 当所有账号都失败过多时

如果当前所有候选账号都已经达到失败阈值，`next()` 不会直接返回空，而是会：

1. 把所有已启用账号的 `failures` 重置为 0
2. 清空 `last_error`
3. 返回第一个账号继续尝试

所以当前实现不是“所有号一旦打坏就整体停机”，而是会自动重置失败计数再试一轮。

---

## 6. 聊天请求入口：`/v1/chat/completions`

这是本项目最核心的流量入口。

### 6.1 请求进入后的标准步骤

当客户端请求 `/v1/chat/completions` 时，服务端大致会做这些事：

1. 读取 `messages` / `model` / `stream` / `tools`
2. 记录基础统计 `_increment_stats()`
3. 取账号
4. 解析/构造会话状态 `chat_id` / `parent`
5. 把 OpenAI 风格消息转换成 Rita 风格消息
6. 如果带 tools，则把工具说明注入最后一条用户消息
7. 组装 Rita 请求头
8. 如果没有现成 `chat_id`，先自动创建一个会话
9. 调上游 `/aichat/completions`
10. 成功后标记账号成功、扣点、记 usage log
11. 再把 Rita SSE 转回 OpenAI Chat Completions 格式

---

## 7. `/v1/chat/completions` 的选号分支

### 7.1 默认分支：走账号池轮换

默认情况下，`/v1/chat/completions` 会：

- 调 `acm.next()` 选一个账号
- 再用 `acm.upstream_headers(acc, RITA_ORIGIN)` 生成上游请求头

请求头主要包括：

- `token`
- `Cookie: token=...`
- `visitorid`（如果该账号有）
- `Origin`
- `Referer`

这也是 Rita 上游鉴权的关键。

### 7.2 特殊分支：客户端自己传 token

`/v1/chat/completions` 还有一个特殊分支：

- 如果请求头里直接带了 `token`
- 服务端会优先使用客户端提供的 token / visitorid
- 此时不会使用轮换出来的账号头作为上游鉴权头

也就是说，当前 `/v1/chat/completions` 支持“客户端自带 Rita token 直通上游”的覆盖模式。

但要注意：

- 代码里虽然仍会先执行 `acm.next()`
- 真正发上游时使用的是客户端给的 token
- 这一分支不等于完全绕过账号池逻辑，只是绕过了“使用账号池 token 作为鉴权头”这一步

### 7.3 与 `/v1/responses` 的区别

`/v1/responses` 当前没有这条“客户端自带 token 覆盖”分支，它始终使用账号池里轮换出的账号。

这是两个接口当前实现上的一个真实差异。

---

## 8. 会话状态如何影响后续轮换请求

### 8.1 为什么需要 conversation state

Rita 的聊天接口不只是发一串 messages，还依赖：

- `chat_id`
- `parent`

如果没有有效 `chat_id`，服务端需要先调：

- `/chatgpt/newConversation`

### 8.2 本项目如何维护会话状态

项目在内存里维护 `_conversation_state`：

- key：由历史消息（除最后一条）摘要后计算出的 hash
- value：
  - `chat_id`
  - `parent`
  - `last_updated`

### 8.3 多轮对话的处理方式

处理逻辑是：

1. 请求进来时，通过消息历史算一个 conversation key
2. 先查内存里有没有旧的 `chat_id` / `parent`
3. 如果没有，自动创建 Rita conversation
4. 上游流返回过程中，抓到 assistant 的消息 id
5. 用新的 `assistant_msg_id` 更新 `parent`
6. 下轮同一上下文再来时，继续沿用这个状态

### 8.4 这个状态的边界

当前 `_conversation_state` 是：

- **纯内存态**
- 进程重启就丢
- 没有持久化到 SQLite

所以：

- 服务重启后，多轮上下文映射会丢失
- 之后请求会重新创建 conversation

这对账号轮换的影响是：

- 账号轮换本身仍正常
- 但 conversation 连续性不保证跨重启保留

---

## 9. 消息格式转换

### 9.1 OpenAI -> Rita

进入 `/v1/chat/completions` 后，项目会把 OpenAI 格式的 `messages` 转成 Rita 需要的 `[{type:"text", text:"..."}]` 结构。

规则大致是：

- `system` 消息会被合并到第一条文本前面，包进 `<system>...</system>`
- `user` / `assistant` 文本会提取成 Rita 的 `text`
- 多模态消息里图片块当前会被压缩成 `[image]`

### 9.2 model 解析

客户端传进来的 `model` 不一定已经是 Rita 的 `model_xxx`。

服务端会先走 `resolve_rita_model()`：

- 如果已经是 `model_xxx`，直接透传
- 如果是别名，如 `rita`、`rita-pro`，会转换成 Rita 内部模型 id
- 如果模型目录缓存里能匹配名称，也会映射
- 最后再不行就原样交给上游处理

这一步决定了后面扣点会按哪个 `model_xxx` 走。

---

## 10. Tool Calling 分支

### 10.1 请求进入时的工具处理

如果 `/v1/chat/completions` 请求里带了 `tools`：

1. 服务端不会直接把 OpenAI tools 原样转给 Rita
2. 而是把工具定义压缩成一段 prompt
3. 注入到最后一条用户消息中
4. 让 Rita 按约定输出 JSON 形式的工具调用意图

### 10.2 响应回来的处理

非流式场景下，如果上游返回的文本里被解析出类似：

- 单工具调用 JSON
- 多工具调用 JSON

服务端会把它再封装成 OpenAI 风格的 `tool_calls` 返回给客户端。

### 10.3 对账号轮换的关系

工具调用本身不改变账号轮换算法；它只是聊天请求进入 Rita 前后的消息包装分支。

也就是说：

- 选号逻辑不变
- 扣点逻辑不变
- 成败统计逻辑也不变

---

## 11. 上游请求真正发出去时的载荷

发往 Rita 上游 `/aichat/completions` 的 payload 关键字段包括：

- `model`
- `messages`
- `online=0`
- `model_type_id=0`
- `chat_id`
- `parent`

这里最关键的，是：

- `messages` 已被转成 Rita 格式
- `chat_id` 可能来自内存会话态，也可能现场新建
- `parent` 由上一轮 assistant 消息 id 决定

---

## 12. 成功、失败、扣点、统计怎么记

### 12.1 何时记成功

在 `/v1/chat/completions` 和 `/v1/responses` 里，代码是在：

- 上游请求成功建立
- 没有出现显式 JSON 错误
- `resp.raise_for_status()` 通过

之后，就立刻执行：

- `acm.mark_ok(acc)`
- `acm.deduct_quota(acc.id, cost)`
- `db.log_usage(...)`

### 12.2 扣点规则

扣点使用 `quota.py` 里的静态 map：

- `get_cost(model_xxx)`

不同模型对应不同点数。

### 12.3 usage 统计

当前 usage 不是精确 tokenizer，而是：

- 按字符数 `/ 4` 近似估算 tokens

记录到：

- `usage_log` 表

### 12.4 一个实现细节

当前实现里，**扣点和 mark_ok 发生在真正消费完上游 SSE 之前**。

这意味着：

- 只要上游连接建立成功并通过前置检查，就会先记成功、先扣点
- 如果后续 SSE 流在中途断掉，当前实现不会把这次已扣的点数自动补回

这是理解运行态表现时一个很重要的细节。

## 13. 流式与非流式返回如何影响链路

### 13.1 Rita 上游本质是 SSE

即使客户端请求的是非流式，Rita 上游仍然是流式 SSE 返回。

所以本项目做了两层处理：

- **stream=true**：边读 Rita SSE，边转成 OpenAI ChatCompletion chunk SSE
- **stream=false**：先把 Rita SSE 全部收集完，再拼成普通 JSON 返回

### 13.2 流式返回时的会话更新

在流式场景下，服务端会边读边捕获 Rita assistant 消息 id。

当流结束后，会调用：

- `update_conversation_state(messages, captured_msg_id, chat_id)`

这样下一轮同上下文请求就能延续 parent 链。

### 13.3 非流式返回时的会话更新

非流式时也会做同样的捕获和更新，只是先把内容攒完，再一次性返回。

---

## 14. `/v1/responses` 如何走账号池

### 14.1 总体逻辑

`/v1/responses` 与 `/v1/chat/completions` 的思路很像：

1. 先把 Responses API 的 `input` 转成 messages
2. 从账号池里取一个账号
3. 自动创建 Rita conversation
4. 调 `/aichat/completions`
5. 扣点并记录 usage
6. 把 Rita SSE 再包装成 OpenAI Responses API 语义

### 14.2 与 `/v1/chat/completions` 的关键差异

主要差异有：

1. **没有客户端 token override 分支**
   - 始终使用账号池选出的账号
2. **Responses API 的返回格式不同**
   - 会包装成 `response.created`
   - `response.in_progress`
   - `response.output_text.delta`
   - `response.completed` 等事件/结构
3. **conversation state 没有像 chat/completions 那样复用 parent 链**
   - 当前这里默认是新建会话后 `parent="0"`

所以如果你把它看成“另一个外壳”，也可以；但底层选号、扣点、上游请求仍然共享同一套账号池。

## 15. 失败处理与账号状态演化

### 15.1 普通失败

当接口调用上游失败时，常见路径会执行：

- `acm.mark_fail(acc, error)`

效果是：

- `failures + 1`
- `total_fail + 1`
- `total_requests + 1`
- `last_error` 更新

此时账号不会立刻被禁用，只是失败计数增加。

### 15.2 达到失败阈值后的效果

只要 `failures >= 3`，该账号就会在 `next()` 里被跳过，不再参与正常轮换。

但它并不是永久下线：

- 如果所有账号都达到阈值，系统会整体 reset failures 再试
- 或者健康检查、手动操作也可能重新恢复它

### 15.3 401 的特殊处理

后台健康检查线程若发现某账号请求上游返回 401，会直接：

- `token_valid=0`
- `enabled=0`
- `disabled_reason='token_expired_401'`

这类账号之后不会再被 `next()` 选中。

### 15.4 手动刷新 token 后的回归

如果该账号支持邮箱重登录刷新 token，刷新成功后可重新：

- 设置新 token
- 清 failures
- 重新启用
- 重新进入轮换池

## 16. 健康检查与轮换池的闭环关系

服务启动后会自动拉起 Token 健康检查线程。

它的职责不是直接服务一次聊天请求，而是做“池子保养”：

1. 定时遍历已启用账号
2. 调 `test_account()` 访问上游模型目录
3. 成功则清 failures
4. 失败则累积失败信息
5. 如果是 401，直接自动禁用

因此整个闭环是：

- **注册模块**：负责把新号放进池子
- **聊天接口**：负责消耗池子里的号
- **健康检查**：负责把坏号标出来或踢出去
- **自动补号**：当活跃号不够时继续补充新号

这四个模块共同构成账号供应链闭环。

---

## 17. 除聊天外，哪些接口也会走同一套轮换池

虽然本文重点是聊天请求，但实际上以下接口也会调 `acm.next()`：

- `/v1/models`
- `/v1/tools`
- `/v1/tools/execute`
- `/v1/conversations`
- `/v1/chat/init`
- `/v1/chat/title`
- `/v1/chat/completions`
- `/v1/responses`

所以账号池并不是只给聊天正文用，而是整个 OpenAI 兼容代理层的统一上游账号分发器。

## 18. 当前实现里的关键注意点

### 18.1 新账号入池后，不保证立刻下一条就轮到它

因为轮换顺序取决于：

- `id` 排序
- 当前 `_robin_index`

所以新号进入池后只是“立刻具备资格”，不是“立刻下一条请求必定命中”。

### 18.2 `responses` 和 `chat/completions` 的分支不完全一致

尤其是：

- `/v1/chat/completions` 支持客户端 header token 覆盖
- `/v1/responses` 当前没有

### 18.3 扣点发生得偏早

当前实现是在上游连接前置通过后先扣点、记成功，再去完整消费 SSE。

所以如果你在排查“为什么明明回复半截断了，点数还是扣了”，需要先想到这一点。

### 18.4 conversation state 是内存态，不跨重启

所以服务重启后：

- 会话 parent 链丢失
- 可能重新 newConversation
- 这不影响账号轮换，但会影响上下文连续性

### 18.5 健康检查才会主动禁用 401 号

普通聊天请求中的错误只会累计 failures，不会在当次请求里直接 `enabled=0`。

也就是说，坏号真正被“自动踢出池子”的关键动作，多数发生在健康检查线程里，而不是聊天主请求里。

---

## 19. 最终总结

注册完成后的账号，并不是简单地存在数据库里等着看，而是会立刻进入以下闭环：

1. 注册成功
2. 入库账号池
3. 立即验活
4. 被 `/v1/chat/completions` 或 `/v1/responses` 轮换选中
5. 作为上游 Rita 请求的 `token` / `Cookie` / `visitorid` 来源
6. 请求成功则清 failures、扣点、记 usage
7. 请求失败则累加 failures
8. 健康检查定期淘汰 401 失效号
9. 活跃账号不足时自动补号继续注册

因此可以把它理解成：

**注册模块负责“生产账号”，账号池负责“分发账号”，聊天接口负责“消耗账号”，健康检查和自动补号负责“维护账号供给稳定”。**
