# Rita2API 账号体系总览

## 1. 文档目的

本文档是本项目账号体系相关文档的总入口，用来帮助快速理解：

- 账号是怎么注册出来的
- 注册后的账号是怎么进入聊天轮换池的
- 健康检查、自动补号、刷新 token 是怎么维护账号池稳定性的
- 遇到不同问题时，应该先看哪一份文档

如果你第一次接触这个仓库，建议先看这份总览，再按阅读顺序进入各子文档。

---

## 2. 一句话理解账号体系

本项目围绕 Rita 账号维护了一个完整生命周期：

**注册造号 → 入库入池 → 聊天请求轮换消耗 → 健康检查淘汰坏号 → 自动补号补充供给 → refresh token / reactivate 修复旧号。**

也就是说，账号体系不是单点功能，而是一整套“账号供应链”。

```mermaid
flowchart TD
    A[手动注册 or 自动补号] --> B[创建临时邮箱]
    B --> C[YesCaptcha and Gosplit 注册链]
    C --> D[token and ticket]
    D --> E[激活 Rita chat 侧]
    E --> F[写入 SQLite accounts]
    F --> G[进入账号池]
    G --> H[/v1/chat/completions]
    G --> I[/v1/responses]
    H --> J[mark_ok or mark_fail and quota]
    I --> J
    J --> K[健康检查线程]
    K -->|401 or 持续失败| L[禁用 or 降级]
    L --> M[active 数下降]
    M --> N[自动补号线程]
    L --> O[refresh token or reactivate]
    O --> G
    N --> G
```

## 6. 账号生命周期总表

### 6.1 阶段 1：创建

来源：

- 手动注册
- 自动补号

结果：

- 账号获得 `token`
- 账号可获得 `ticket`
- 账号写入 `accounts` 表
- 默认 `enabled=1`
- 默认 `quota_remain=100`

### 6.2 阶段 2：进入轮换池

条件：

- `enabled=1`
- `quota_remain > 0`
- `failures < 3`

结果：

- 可被 `AccountManager.next()` 选中
- 可承接模型目录、聊天、工具、会话等上游流量

### 6.3 阶段 3：被聊天请求消耗

行为：

- 被 `/v1/chat/completions` 或 `/v1/responses` 选中
- 用该账号的 `token` / `Cookie` / `visitorid` 请求 Rita
- 请求成功时：
  - `mark_ok`
  - 扣点
  - 记录 usage
- 请求失败时：
  - `mark_fail`
  - failures 增加

### 6.4 阶段 4：退化或失效

可能原因：

- 普通请求失败过多
- 健康检查命中 401
- quota 归零
- 手动禁用

表现：

- 失败号被 `next()` 跳过
- 401 号会被后台健康检查自动禁用
- quota 为 0 的号不会再进入候选列表

### 6.5 阶段 5：修复或替换

修复路径：

- refresh token
- reactivate
- relogin for ticket

替换路径：

- 自动补号
- 手动继续注册新号

结果：

- 旧号被修回池子
- 或新号补进池子

## 7. 从“问题”反推应该看哪份文档

### 7.1 注册按钮点了没反应 / 注册失败

先看：

- [注册整体流程](./注册整体流程.md)

重点关注：

- 配置检查是否通过
- `YESCAPTCHA_KEY` 是否存在
- 默认邮箱渠道是否配置完整
- OTP 是否收到
- token/ticket 是否成功拿到

### 7.2 为什么新号注册成功了，但聊天没走到它

先看：

- [注册后聊天请求轮换流程](./注册后聊天请求轮换流程.md)

重点关注：

- `next()` 的候选条件
- round-robin 顺序不是按注册时间，而是按 `id` 排序 + `_robin_index`
- 新号只是“立刻有资格”，不代表“下一条一定命中”\

### 7.3 为什么有账号被踢掉 / 活跃数突然下降

先看：

- [健康检查、自动补号、刷新 Token 联动流程](./健康检查自动补号刷新Token联动流程.md)

重点关注：

- 后台健康检查是否命中 401
- failures 是否累计过高
- 是否因为账号被禁用导致 active 下降

### 7.4 为什么系统又开始自动补号了

先看：

- [健康检查、自动补号、刷新 Token 联动流程](./健康检查自动补号刷新Token联动流程.md)

重点关注：

- `summary().active` 是否低于 `AUTO_REGISTER_MIN_ACTIVE`
- 当前代码是否真按 quota 补号（答案是：当前不是）

### 7.5 为什么点数扣了但回复中途断了

先看：

- [注册后聊天请求轮换流程](./注册后聊天请求轮换流程.md)

重点关注：

- 当前实现里 `mark_ok` 和 `deduct_quota` 的时机偏早
- 它发生在完整消费上游 SSE 之前

## 9. 当前实现的几个核心事实

### 9.1 自动补号按 active 数，不按总 quota 数

虽然配置表里有 `AUTO_REGISTER_MIN_QUOTA`，但当前代码里并没有真正把它接入自动补号判断。

### 9.2 手动 `health-check/run` 与后台健康检查线程不完全等价

后台线程的 auto-disable 会真正写库；手动 health-check/run 当前只是返回统计，不会真正把 401 禁用状态持久化。

### 9.3 refresh 成功相当于“完整复活账号”

不仅换 token，还会：

- 重新启用
- 清 failures
- 清 disabled_reason
- 重置 quota 到 100

### 9.4 conversation state 是内存态

它影响多轮会话连续性，但不持久化到 SQLite；服务重启后会丢。
